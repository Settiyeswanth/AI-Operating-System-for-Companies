6. Enterprise Memory Architecture

The system implements a three-tier memory model — a design pattern borrowed from cognitive architecture research and implemented in production by OpenAI's memory systems and Microsoft's Recall platform.

TierStoreContentQuery TypeAccess Pattern
Episodic SQLite Event Ledger Append-only log of every NormalizedEvent; never overwritten Temporal ("what happened between dates X and Y") LedgerWriter.append(); audit replay only
Semantic Neo4j Knowledge Graph Entities, relationships, decisions, org structure Traversal ("who worked on X", "what constrains Y") GraphClient.find_*() — always confidence-filtered
Associative Qdrant Vector Store Embedded document chunks (summaries, PR descriptions, comments) Semantic similarity + BM25 hybrid VectorClient.hybrid_search() — access-control filtered

6.1 Hybrid Retrieval: Reciprocal Rank Fusion

The vector client implements RRF fusion (score(d) = Σ 1/(k + rank(d, list))) to combine dense ANN search and BM25 sparse search. This is architecturally correct: organizational content has a bimodal retrieval profile — exact entity names (e.g. "PROJ-412", "payment_service.py") are served by BM25; semantic equivalences (e.g. "auth" → "authentication service") are served by dense vectors. Neither method alone achieves adequate recall on enterprise corpora.

6.2 Privacy-by-Design: Message Storage Invariant

Critical invariant: Raw message content is NEVER stored in the Knowledge Graph or Vector Store. The Message entity stores only an LLM-generated content_summary. This is enforced at the enrichment stage before any write to persistent storage. Violating this invariant constitutes a privacy and compliance failure.

6.3 Confidence Threshold Architecture

Two confidence thresholds govern what information reaches users:

  • Edge confidence threshold: 0.70 — Graph edges below this score are stored but never included in query answers. LLM-inferred edges are seeded at 0.60 and require human confirmation or additional corroborating events to cross the threshold.
  • ER auto-merge threshold: 0.85 — Entity resolution merges below this confidence are held for human review. A wrong merge is worse than a missed merge: it produces incorrect answers without any error signal.

7.3 MonitorAgent — Rule Engine Architecture

Phase 1 ships exactly two rules, chosen for high precision over recall. Alert fatigue is the primary failure mode for a proactive monitoring agent — adding low-precision rules produces alert blindness before the product demonstrates value.

Rule IDDescriptionGraph QueryP1 ThresholdAlert Type
blocked_without_decision_v1 Feature has status=blocked for > N days with no CONSTRAINS Decision MATCH (f:Feature) WHERE f.status='blocked' AND NOT (f)<-[:CONSTRAINS]-(:Decision) 5 days (HIGH if >10) BLOCKED_WITHOUT_DECISION
completion_mismatch_v1 Feature is SHIPPED but linked Requirement is still OPEN MATCH (f:Feature)-[:IMPLEMENTS]->(r:Requirement) WHERE f.status='shipped' AND r.status='open' Always HIGH COMPLETION_MISMATCH
Alert deduplication (OPEN ITEM): The current MonitorAgent fires a new alert every poll cycle for the same condition. A Redis SET NX EX 86400 key keyed on rule_id:feature_id must be added before Milestone 4 to prevent alert spam. Without this, the first 24-hour run will produce hundreds of duplicate alerts, destroying user trust in the alert system permanently.

9. Connector & Ingestion Architecture

9.1 Connector Design Principles

  • Connectors produce only RawEvent objects. Zero business logic. Schema mapping happens in the ingestion service.
  • Webhook signature validation is mandatory before emitting any event. Rejection rate is logged; 5-minute tolerance on timestamp checks.
  • Polling is primary in Phase 1. Webhooks are supplementary real-time delivery. Historical backfill is a separate operation.
  • Dead-letter queue for all failures. Connectors never crash on a single bad event — they log and continue. Failed events go to aios:dlq Redis list.
  • Idempotency key is set by the connector using the canonical format: source_system:event_type:entity_id:timestamp.iso(). Ingestion deduplicates on this key.

10. Entity Resolution Pipeline

Entity Resolution is the highest-risk component in the system. A missed merge (split entity) produces incomplete answers. A wrong merge (false merge) produces incorrect answers and may constitute a privacy violation. The system is biased toward splits over merges at every decision point.

Stage 1: Deterministic (confidence = 1.0)
  email present in event?
    YES → lookup by canonical_email (case-normalized)
          found → map source_id → canonical_id
          not found → create new canonical Person, map
    NO → Stage 2

Stage 2: Probabilistic (confidence = computed score)
  weighted score = 0.4 × email_domain_match
                 + 0.3 × normalized_levenshtein(display_name)
                 + 0.2 × username_pattern_match
                 + 0.1 × temporal_co_occurrence

  score ≥ 0.85 → auto-merge (log with confidence)
  score < 0.85 → HELD: push to review_queue (SQLite)
                  event.processing_status = HELD
                  event NOT forwarded to enrichment
                  await human resolution via er_review.py CLI
Critical negative test: Two people with similar names but different email domains must NEVER be auto-merged. The test test_different_emails_create_different_persons in the test suite covers this exact scenario and must never be removed or relaxed.

11. Security Model

11.1 Authentication

PhaseMechanismImplementation
Phase 1Bearer token (API key)Any non-empty Bearer token accepted; token IS the user_id. Scopes default to ["*"]. Simple but functional for prototype.
Phase 2OAuth2 + RBACKong/Traefik + JWT with team-scoped claims. Scopes: team:engineering, team:product, etc. PII grants explicit per-user.

11.2 Authorization — Runtime Access Control

Access control is enforced at read time on every retrieval, never at write time. The AccessTags.is_accessible_by() method is called on every RetrievedChunk and every graph node before it reaches the agent. Excluded artifacts are never disclosed in responses.

  • min_scope: user's scopes must include this scope to see the artifact (e.g. "team:engineering")
  • restricted_to: explicit allowlist of user IDs; if set, only listed users can access
  • pii_flag: requires explicit "pii" grant in user_grants; not derivable from team membership
  • sensitivity_level: public | internal | confidential | restricted (informational, not enforced at P1)

11.3 Data Privacy Invariants (non-negotiable)

  1. Raw message content from Slack/Linear comments is NEVER written to Neo4j or Qdrant — only LLM-generated summaries.
  2. PII-flagged entities are excluded from query results unless the requesting user has an explicit pii grant.
  3. Excluded artifacts are never disclosed in error messages or source lists — their existence is not revealed.
  4. All ER merge decisions are logged with the source event ID — every merge can be audited and reversed.
  5. LLM providers (Ollama in Phase 1) run locally — no organizational data leaves the deployment boundary.

17. Technical Risks & Mitigations

RiskProbImpactMitigation
Alert fatigue from high false positive rate on MonitorAgent High High Ship ONLY 2 rules in P1. Measure precision from day 1. Add rules only after ≥60% confirmed precision on existing rules. No exceptions.
Alert deduplication missing — same alert fires every 5-min cycle High High Add SETNX EX 86400 Redis key per (rule_id + feature_id) before Milestone 4. Treat as blocker for Milestone 4 sign-off.
Ollama latency too high for interactive queries (8B model on CPU) High Medium Use llama3.1:8b for all ops. Cache frequent query embeddings in Redis (TTL 1hr). Route verification to smaller/faster model. Set 30s timeout on gateway.
Entity resolution splits same person across connectors High High Email-based deterministic resolution covers ~80% of cases. Instrument ER precision from day 1 of data ingestion. Weekly sample audit of top-10 review queue items.
gateway + enrichment services not yet scaffolded (docker-compose references them) High High Create minimal Dockerfile + FastAPI skeleton for gateway. Create enrichment pipeline (subscribe to aios:enrichment → embed → write to Neo4j/Qdrant). Blocks Milestone 2.
LangGraph state machine hangs on LLM timeout Medium High Per-node asyncio timeout of 30s (wrap each LLM call). Circuit breaker on LLMGateway: 3 consecutive failures → return UNCERTAIN verdict, log alert. Never hang.
Neo4j graph queries slow on large traversals Low Medium Max 4 hops enforced in queries (GraphClient.max_retrieval_hops). Key property indices created at seed time. EXPLAIN PROFILE on any new Cypher query before merge.
LLM cross-reference extraction introduces spurious graph edges Medium Medium All LLM-inferred REFERENCES edges seeded at confidence=0.6 (below answer threshold of 0.7). These edges never appear in cited answers without human confirmation.
BM25 index lost on enrichment service restart during active ingestion Medium Low Add startup loader: scan Qdrant collection → rebuild BM25 corpus. Log corpus size at startup. Phase 2: persist BM25 index to Redis Sorted Set.
Slack webhook signature validation fails due to replay/clock skew Medium Low Use 5-minute tolerance window on timestamp check. Log all validation failures with headers. Never silently accept invalid signatures.
